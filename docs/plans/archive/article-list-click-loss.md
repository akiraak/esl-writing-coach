# 記事一覧のクリックがときどき効かない問題の修正

## 目的・背景

「新規作成や削除ボタンを押しても反応しないときがある。記事を選択しても反応しないときがある」
という報告。調査の結果、`loadArticles()` が呼ばれるたびに `listEl.innerHTML = ''` で
**記事一覧の DOM を全再構築**していることが原因と判断した。

- `loadArticles()` は自動保存の完了ごと（入力停止の 2 秒後）・flushPendingSave 後・削除後など
  頻繁に走る
- クリック（mousedown → mouseup）の途中で押していた要素が破棄されると、click イベントは
  発火しない（共通祖先の ul がターゲットになり、li のリスナーに届かない）
- ユーザーが本文を書いた直後に一覧を操作すると、ちょうど自動保存 → 一覧再構築のタイミングと
  重なりやすく、「ときどき反応しない」症状と一致する
- 「新規作成」ボタン自体は再構築されないが、症状の主因（一覧側）と合わせて報告されたもの。
  API 不達時の無言失敗は前回修正済み（エラー表示あり）

## 対応方針

`loadArticles()` を全再構築から **id ベースの差分更新** に変更する。

- 既存の li を `dataset.id` で Map に取り、あれば使い回して excerpt / 日付 / active だけ更新
- 無い記事だけ新規に li を作成（クリック・削除のリスナーは作成時に一度だけ付与）
- 並び順は先頭からカーソル比較し、位置が合っている li は動かさない
  （`insertBefore` は同一位置なら no-op にする。ノードを動かすと同じくクリックが失われるため）
- 一覧から消えた記事の li だけ remove する

これにより通常の自動保存サイクル（選択中の記事の excerpt / 日付が変わるだけ）では
DOM ノードが一切差し替わらず、クリックが失われなくなる。

## 影響範囲

`backend/public/js/app.js` の `loadArticles()` 周りのみ。HTML / CSS / バックエンドは変更なし。

## テスト方針

ヘッドレス Chromium で:
1. `loadArticles()` を連続で呼んでも li の要素が同一（identity 維持）であること
2. 記事内容の更新（PUT）→ `loadArticles()` で同じ li のまま excerpt / 日付が更新されること
3. 記事選択・削除・新規作成の通常フローが従来どおり動くこと（検証で作った記事は削除して後始末）
