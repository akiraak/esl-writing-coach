# 削除確認の独自ダイアログ化 + 失敗理由の表示

## 目的・背景

モバイルでの報告 2 件への対応。

1. 「記事作成ボタンで『記事の作成に失敗しました』が出る」— ローカルサーバのログに失敗の
   痕跡がなく、こちらでは再現しない。原因特定のため、エラーメッセージに理由
   （HTTP ステータス or 通信エラー）を付けて表示する
2. 「削除の確認に『ダイアログを表示しない』が出る。OK を押しても消えない」—
   ブラウザ標準の `confirm()` は、連続表示するとブラウザが抑止チェックボックスを付ける。
   さらに一度チェックされると以後 confirm が常に false を返し、削除が無反応になる罠がある。
   また削除の fetch 失敗が無言なため、失敗 → 再タップ → 再ダイアログが「OK しても消えない」
   ように見えていた可能性が高い

## 対応方針

- `confirm()` をやめ、ルールダイアログと同系の `<dialog>`（`#confirm-dialog`）で
  「削除する / キャンセル」を出す。Esc・backdrop クリックはキャンセル扱い
- 削除の fetch に res.ok チェック + try/catch を追加し、失敗時は
  「削除に失敗しました（HTTP xxx / 通信エラー）」をステータス欄に表示
- 新規作成の失敗表示にも同じ理由付けを追加（「記事の作成に失敗しました（…）」）

## 影響範囲

- `backend/public/index.html` — 確認ダイアログの追加
- `backend/public/css/style.css` — `.confirm-dialog` のスタイル
- `backend/public/js/app.js` — confirmDialog()・削除/新規作成のエラーハンドリング
- バックエンドは変更なし

## テスト方針

ヘッドレス Chromium で: 削除タップ → 独自ダイアログ表示 → キャンセルで消えない /
削除するで記事が消える、DELETE・POST をブロックした状態でそれぞれのエラーメッセージに
理由が付くこと、通常の新規作成フローが従来どおり動くこと（検証記事は後始末）。
