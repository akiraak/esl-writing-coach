# 「保存に失敗しました」の診断力強化 + 保存の自動リトライ

## 目的・背景

本番（esl-writing.chobi.me）で「保存に失敗しました」の報告。本番 DB を確認すると、
対象記事は作成 → 保存（PUT 22:30:16）→ 添削（22:30:20）まで一度は全て成功しており、
その後の保存だけが失敗している。サーバログにエラー・再起動はなし。

状況からは「Access セッション失効（Session Duration なし）+ ブラウザが
セッションリカバリ導入前の古い app.js のまま」が濃厚だが、現状の「保存に失敗しました」
には理由が無く、サーバ側にも 4xx/5xx の記録が残らないため確定できない。

## 対応方針

1. **フロント: エラー理由の表示を保存・添削にも拡大**（`app.js`）
   - `failReason()` を拡張: TypeError → 通信エラー、それ以外は err.message（60 字まで）
   - save() の throw を `HTTP xxx` 形式にし、「保存に失敗しました（HTTP xxx / 通信エラー）」
   - runCorrection() も「添削に失敗しました（…）」（502 の body.error はそのまま理由に出す）
2. **フロント: 保存の自動リトライ**（`app.js`）
   - save() 失敗時に 10 秒後の再試行を予約（現状は次の入力まで未保存のまま放置される）
   - ステータスは「保存に失敗しました（…）10 秒後に再試行します」
3. **バックエンド: 非 2xx レスポンスのログ**（`index.ts`）
   - `res.on('finish')` で status >= 400 のとき `method path -> status` を console.log
   - 本番で「保存が 404/400/413 だった」等がコンテナログから追えるようになる

## 影響範囲

`backend/public/js/app.js`（表示とリトライ）、`backend/src/index.ts`（ログ 1 ミドルウェア）。
API 挙動・DB は変更なし。

## テスト方針

ヘッドレス Chromium で: PUT を 500 に偽装 → 「保存に失敗しました（HTTP 500）…」表示 +
10 秒後に再試行されること、PUT 復旧後にリトライで保存されること。
バックエンドは 404 を返すリクエストでログ行が出ることを確認。typecheck。
