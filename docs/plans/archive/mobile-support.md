# モバイル対応（スマホからの利用）

## 目的・背景

esl-writing.chobi.me を iPhone 等のスマホから実用的に使えるようにする。

現状をモバイル幅（390x844、ヘッドレス Chromium）で確認したところ、以下の状態だった。

- サイドバー（240px）+ ディバイダー（20px）が常時表示され、エディタ領域が約 130px に潰れる。
  エディタのペインは画面右へはみ出して見切れる（横スクロールが発生）
- 記事を選択しても編集欄・出力欄がまともに読めず、実質的にスマホでは使えない

### 現状の実装と問題点（調査結果）

1. **レイアウト**: `.layout` は常に flex 横並び。`@media (max-width: 900px)` は
   `--sidebar-w: 240px` への縮小と `editor-grid` の 1 カラム化のみで、
   サイドバーとエディタの同時表示をやめる仕組みがない
2. **開閉ボタン**: `.sidebar-divider` は幅 20px の縦長ボタンで、タッチでは押しづらい
   （タップターゲットの目安 44px に遠く及ばない）
3. **開閉状態の復元**: localStorage（`sidebarCollapsed`）に保存されるため、
   スマホ初回アクセスではサイドバーが開いた状態で始まり、エディタが潰れた画面が最初に見える
4. **ビューポート高さ**: `.layout` と `.rules-dialog` が `100vh` 基準。
   モバイル Safari のアドレスバー伸縮・仮想キーボードで実高さとずれる（`dvh` 未使用）
5. **フォントサイズ**: textarea が 15px。iOS Safari は 16px 未満の入力欄フォーカス時に
   ページを自動ズームするため、書くたびに画面が拡大される
6. **ルールダイアログ**: `width: calc(100vw - 80px)` / `height: calc(100vh - 80px)` で
   スマホでは余白が過大、かつキーボード表示時に textarea が隠れる
7. **タップターゲット**: 記事一覧の「削除」（`.btn-small`、padding 3px 10px）が小さい
8. **topbar**: `.status` の `min-width: 8em` が狭い画面で h1 と競合する

サーバ側（Express / SQLite / 添削 API）はレスポンシブ対応と無関係で、変更は不要。
自動保存 → 添削のフローはタッチでもそのまま動く。

## 対応方針

ブレークポイント **640px 以下をモバイル** とし、既存の 900px（editor-grid 1 カラム化）は
そのまま残す。デスクトップの見た目・挙動は変えない。

### Phase 1: サイドバーのドロワー化（モバイルレイアウトの本体）

- モバイルではサイドバーを **オーバーレイのドロワー** にする
  - topbar 左端にハンバーガーボタン（☰）を追加（モバイルのみ表示）
  - 開くと左から `min(85vw, 320px)` のドロワー + 背景ディム（タップで閉じる）
  - 記事選択・新規作成でドロワーを自動で閉じてエディタに切り替える
- `.sidebar-divider` はモバイルでは非表示（デスクトップ専用のまま）
- localStorage の開閉状態は **デスクトップのみ** に適用（モバイルは常に「閉」で開始し、
  記事未選択時のみ初期表示でドロワーを開く）
- 既存の `.sidebar.collapsed`（width: 0）とドロワー用の開閉クラスは分離し、
  デスクトップ⇔モバイルのリサイズで状態が混ざらないようにする

### Phase 2: 入力体験・細部の調整

- `100vh` → `100dvh`（`height: calc(100vh - 57px)` を先に書き、`dvh` で上書きする
  フォールバック形式。`.layout` と `.rules-dialog` の両方）
- textarea / `.output` の font-size を 15px → **16px** に統一（iOS の自動ズーム抑止。
  デスクトップも 16px にして分岐を作らない）
- ルールダイアログはモバイルでほぼ全画面（`calc(100vw - 16px)` × `calc(100dvh - 16px)` 程度）
- タップターゲット拡大（モバイルのみ）: 記事一覧の「削除」ボタンの padding 増、
  記事行の padding 見直し
- topbar: モバイルでは `.status` の `min-width` を外し、h1 のテキストを省略可能にする
  （`text-overflow: ellipsis`）

### Phase 3: 検証と本番反映

- ヘッドレス Chromium（390x844）で確認:
  ドロワー開閉 / 記事選択でドロワーが閉じてエディタ表示 / ルールダイアログ /
  記事未選択時の初期表示
- デスクトップ幅（1280px 等）で従来と同じ表示・挙動であることを確認（リグレッションなし）
- 本番反映（g3plus 上で pull + rebuild、手順は g3plus-ops の
  docs/workflows/esl-writing-coach.md）後、iPhone 実機で確認

## 影響範囲

- `backend/public/index.html` — ハンバーガーボタン + ドロワー用ディム要素の追加
- `backend/public/css/style.css` — モバイル用メディアクエリ（640px）、ドロワー、
  dvh、フォントサイズ、タップターゲット
- `backend/public/js/app.js` — ドロワー開閉ロジック、記事選択時の自動クローズ、
  localStorage 適用のデスクトップ限定化
- バックエンド（TypeScript / DB / API）は変更なし

## テスト方針

- Phase 3 のとおり。モバイル幅・デスクトップ幅の両方をスクリーンショット付きで確認し、
  最後に iPhone 実機（本番 URL、Cloudflare Access ログイン込み）で確認する
